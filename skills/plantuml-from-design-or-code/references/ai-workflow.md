# 面向 AI 的 UML 执行清单

当任务需要严格执行步骤，而不是只看简版 skill 说明时，读取本文件。

## 输入

最少输入：

- 图的目标
- 目标读者
- 相关源码和/或设计文档
- 输出路径

可选输入：

- 期望图类型
- 事实来源：`design-doc` / `source-code` / `hybrid`
- 描述性文字语言（默认中文，代码标识符保持原名）
- 粒度要求

## 执行清单

1. 用一句话写出这张图要证明的事实。
2. 选择事实来源：
   - `design-doc`
   - `source-code`
   - `hybrid`
3. 选择图类型。
4. 阅读相关设计文档和/或源码。
5. 建事实表：
   - 设计模式：组件、层次、概念流程、约束
   - 源码模式：对象、调用、状态变化
   - 混合模式：设计骨架 + 实现确认
6. 先画主干。
7. 再补 `note` 说明。
8. 检查抽象粒度一致性：
   - 避免在同一流程里混用高层阶段和 helper 级细节
   - 保持节点粒度稳定
9. 检查 `note` 锚点是否正确：
   - 注释是否真的挂在其语义对应的位置
   - 全局说明不要错挂到某个局部步骤
   - 局部说明不要跨阶段挂到别的节点
10. 检查可渲染性。
11. 按选定来源检查一致性。
12. 如果图集已有 README / 索引，必要时更新。

重要规则：

- 默认单次仅输出一种策略下的图示结果。
- 除非用户有明确要求，否则不得自动生成多张不同层级的图示。
- 如果图开始采用某一层粒度，后续就保持这一层粒度。

## 提示模板

```text
Use $plantuml-from-design-or-code.

Goal: <一句话说明要证明的事实>
Audience: <架构教学 / 代码导读 / 调试分析>
Fact source: <design-doc / source-code / hybrid>
Diagram type: <class / activity / sequence>
Inputs:
- <设计文档路径，如有>
- <源码路径 1>
- <源码路径 2>
Output path: <目标 .puml 路径>

Requirements:
1. 先按选定来源抽事实，再画图。
2. 绘图布局排版避免使用曲线关系线（可在图头部声明 `skinparam linetype ortho` 或 `skinparam linetype polyline`）。
3. 代码标识符与源码保持一致。
4. 长说明放 note，不在线上堆长标签。
5. 图内保持抽象粒度一致。
6. 完成前做一致性复核。
```

## 常见失败模式

- 仅凭记忆或主观概念而非源码与文档提取事实
- 把静态结构和动态流程混在一张图里
- 在同一张图内混入高层概念步骤和低层实现细节
- 注释挂载位置只图方便，不符合其真实语义范围
- 将设计概念图偏离为底层实现细节图
- 在实现对照图中随意混入不切实际的设计假设
- 边标签过长
- 分支条件没有对照实现核实
- 使用预览器不兼容的 PlantUML 语法而未做检查
