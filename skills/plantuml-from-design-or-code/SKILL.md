# 从设计或源码生成 UML

基于明确的客观事实生成 PlantUML 图示，严禁凭主观印象绘图。

## 工作流

1. 首先明确定义图示的目标，用简练的语言说明其核心意图：
   - 该图示旨在阐明或证明何种客观事实？
2. 选定事实来源：
   - 设计文档：表达系统架构意图或概念结构
   - 源码：表达当前实现
   - 混合模式：以设计意图为骨架，并结合具体实现进行校验
3. 选择图类型：
   - C4 模型图（上下文/容器/组件）：表达系统级、应用级或模块级的宏观静态边界与依赖。
   - 类图：拥有关系、继承关系、代码级模块边界。
   - 流程图：步骤流程与条件分支。
   - 活动图：带角色泳道与并发分支的复杂业务流程。
   - 序列图：对象之间的交互顺序。
   - 状态图：单一核心实体在不同触发条件下的状态流转。
   - 部署图：软件在物理或虚拟节点上的部署拓扑。
   - 产物依赖图：代码库、构建产物或包模块之间的编译时/运行时依赖关系。
4. 基于选定的事实来源提取要素信息，随后开始绘制图示：
   - 设计文档：组件、层次、概念流程、约束
   - 源码：参与类/结构体、调用顺序、状态变化
   - 混合模式：整合设计骨架与实现确认
5. 优先绘制核心主干结构：
   - 关键节点
   - 关键边
   - 关键分支
6. 详细的说明文字应承载于 note 中，避免在连接线上堆叠长句标签。
7. 最终依据选定的事实来源执行严格的一致性复核。

## 规则

- **【强制约束】编码前预读模板**：在开始生成特定图表之前，必须先读取 `templates/` 目录下对应的 `.puml` 示例文件（如 `sequence_example.puml` 等），严格模仿其排版声明（如 `skinparam linetype ortho`）、注释挂载方式和语法结构，严禁在排版样式上自由发挥。
- 优先采用多张聚焦单一主题的细粒度图示，避免使用单张信息过载的复杂大图。
- 绘图布局：排版避免使用曲线关系线（可在图头部声明 `skinparam linetype ortho` 或 `skinparam linetype polyline`，以保持图面整洁）。
- 代码标识符尽量与源码保持一致。
- 描述性文字（注释、说明、标签）默认使用中文；代码标识符、类名、方法名保持原名。
- 边标签尽量简短：`owns`、`extends`、`dispatch`、`syncs`、`reads`。
- 若属于实现导向的图示，在完成绘制前必须严格核对代码的关键调用顺序。
- 若属于设计导向的图示，禁止私自补充设计文档中未阐明的底层辅助实现细节。
- 若采用混合模式，需清晰界定并区分哪些内容代表设计意图，哪些内容反映当前实际实现。
- 同一张图内保持抽象粒度一致。
  - 不要在同一区域同时混用高层概念步骤和低层 helper / 局部变量细节，除非这种对比本身就是主题。
  - 若图示起始于特定粒度层级，后续节点应保持同一抽象粒度。
  - 当细节内容过于复杂时，应优先简化或拆分图示，严禁在同一图示中混淆不同的抽象层级。
- `note` 必须锚定到语义真正对应的节点或区域。
  - 严禁仅因版面排版便利，将总结性或范围性注释放置在邻近但语义不相关的步骤或节点上。
  - 如果注释描述的是“整张图的入口语义”或“全局流程语义”，优先挂到 `start`、标题附近，或独立的整体说明区域。
  - 若注释描述的是局部步骤、副作用或分支约束，则须将其关联至该局部节点，严禁跨阶段错误挂载。

## 来源模式

### 设计文档模式

适用于：

- 架构教学图
- 系统概念组件图
- 设计意图业务流程图
- 尚未实现或无需绑定具体实现的结构图

此模式下：

- 以设计文档为主事实来源
- 输出保持概念层粒度
- 未包含的实现细节应予以明确省略，严禁凭空臆断

### 源码模式

适用于：

- 物理实现图
- 精确调用关系序列图
- 调试导向控制流程图
- 与当前代码执行路径严格对照的图示

此模式下：

- 源码是唯一权威
- 必要时可深入至辅助函数（helper）级的具体行为
- 条件分支必须与实现一致

### 混合模式

适用于：

- 对比设计意图与当前实际实现
- 首先基于设计文档构建主架结构，随后利用源码进行校验
- 展示实现方案与设计方案的一致性

此模式下：

- 首先基于设计文档构建主架结构
- 再用源码核对关键关系和关键路径
- 若发现设计与实现不一致，须显式声明并指出，严禁采取模糊的静默融合处理

## 校验

完成时必须至少针对以下两项指标进行检查：

1. 可渲染性
   - 没有 PlantUML 语法错误
   - 泛型类名等敏感语法在预览器中可用
2. 一致性
   - 与所选事实来源一致
   - 调用顺序与分支条件须具备真实代码依据，严禁凭空臆造
   - 没有伪造关系

## 绘图排版规范

- **强制使用直角连线**：图表默认必须声明 `skinparam linetype ortho` 保持排版横平竖直。
- **避免线上注释过密**：当连接线上的说明文字过长时，必须将其精简为极短的标签（如动作名称），并将详细说明放入 `note on link` 或节点旁边的 `note` 中，严禁在连线上堆砌长句。

## 常见模式

### C4 模型图 (上下文 / 容器 / 组件)
> 对应模板: `templates/c4_context_example.puml`

- 上下文图仅展示系统与外部用户/外部系统的宏观边界。
- 容器图展示独立部署的技术单元（如微服务、数据库）。
- 组件图展示单个容器内部的高层模块组织。
- **强制约束**：严格保持当前图示的抽象层级，严禁在上下文图中暴露组件级或代码级细节。

### 类图
> 对应模板: `templates/class_example.puml`

- 用 package 表示分层
- 长语义解释放到 `note`
- 除非 API 本身是图示主题，否则无需将所有方法完整列出

### 流程图
> 对应模板: `templates/flowchart_example.puml`

- 仅保留具备说明与展示价值的分支路径
  - 优先保留会影响主流程走向、揭示核心机制，或承载关键知识点的业务分支
  - 禁止将所有实现级别的条件分支（如 if/else）全盘绘制在图示中
  - 对理解主流程无显著帮助的局部优化、日志输出、临时兼容等次要分支默认应予省略
- 若两条执行路径的差异过大，应当拆分为两张图示表达，严禁强行合并

### 活动图
> 对应模板: `templates/activity_example.puml`

- 适用于跨越多个角色（泳道）的复杂业务流或含有并发分支的执行流水线。
- 必须合理使用泳道（Swimlanes）划分模块边界，避免跨角色逻辑不清。

### 序列图
> 对应模板: `templates/sequence_example.puml`

- 适用于需要明确调用关系与消息交互顺序的场景
- 参与者仅保留对系统状态具有关键性改变的对象

### 状态图
> 对应模板: `templates/state_example.puml`

- 适用于生命周期复杂、状态流转严密的核心实体对象。
- 必须明确标注触发状态转换的事件（Event）或动作（Action）。
- 避免过度复杂的网状跳跃，当状态过多时应使用复合状态（Composite State）进行封装。

### 部署图
> 对应模板: `templates/deployment_example.puml`

- 重点展示物理节点（Node）、执行环境（Execution Environment）与应用制品（Artifact）的映射关系。
- 适用于说明网络边界、高可用物理架构或分布式部署形态。

### 产物依赖图
> 对应模板: `templates/dependency_example.puml`

- 重点展示组件包（Package）、构建产物（Artifact）或微服务之间的调用与编译依赖。
- 必须使用构造型（Stereotype，如 `<<executable>>`, `<<shared lib>>`, `<<static lib>>`）严格区分产物的具体物理形态（可执行文件、动态库、静态库等）。
- 必须明确区分“编译时静态链接”与“运行时动态链接/调用”。
- 当发现环形依赖（Circular Dependency）时，需通过高亮或 `note` 显式标识出架构设计风险。

## 什么时候拆图

当出现以下任一情况时，应采取拆图策略：

- 试图在单张图示中同时表达静态结构与动态行为流程
- 标签严重重叠
- 一个以上的主题在竞争读者注意力
- 图示主旨不够聚焦，无法在短时间内直观传达核心意图

## 输出约定

- 输出的图示文件默认存储于项目的 `docs/uml/` 目录中，除非另有明确指定。
- 文件名尽量直接表达主题，例如：
  - `01_system_assembly.puml`
  - `02_core_execution_layers.puml`
  - `05_rob_storebuffer_flow.puml`
- 新增图示文件后，若存在图集索引或说明文档，应当同步更新。
