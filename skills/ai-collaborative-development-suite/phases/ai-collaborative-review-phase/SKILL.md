# 检视阶段

[目标]
针对步长 N 的逻辑、边界条件及与原设计的一致性进行静态审查。

[准入条件]
- 步长 N 的代码实现已完成。
- 必须从 `TASK_STATE.md` 中获取上下文设计。

[退出条件]
- 识别出的问题已修复或被明确记录接收。
- 更新 `TASK_STATE.md`。

[执行步骤]
1. 比对代码实现与设计图示，审查是否存在结构性偏移。
2. 审查极端边界、死代码 (Dead Code) 和重复逻辑 (DRY)。
3. 针对新发现的报错触发自动修复 (最大重试: 2 次)。
4. 审查通过 -> 流转至验证阶段。

[质量底线]
- [必须] 审查重心锁定为：逻辑正确性、异常边界与回归风险。
- [禁止] 陷入“自行车棚效应” (如纠缠于空格、缩进位置、大括号换行等代码格式争论)。
- [禁止] 将阻断级的严重错误与代码风格级别的建议混为一谈。
- [必须] 必须识别出仅为了临时跑通而写入的死代码并标注。

[升级人工条件]
- 发现代码结构与原设计发生了根本性冲突。
- 阻断级错误在 2 次自动修复后依然无法解决。
- 存在不可接受的高风险，必须由人类强行确认放行。
