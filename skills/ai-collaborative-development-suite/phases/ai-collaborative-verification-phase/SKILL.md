# 验证阶段

[目标]
执行测试并收集客观证据，从而判定步长 N 是否已达成验收标准。

[准入条件]
- 步长 N 检视已完成。
- 从 `TASK_STATE.md` 获取明确的验收标准。

[退出条件]
- 已收集测试成功/失败的客观证据。
- 决策结论: 开启下一小步 或是 结束总任务。
- 更新 `TASK_STATE.md` (若结束总任务则需触发归档清理)。

[执行步骤]
1. 针对步长 N 执行其专属的测试脚本。
2. 汇总提取输出日志/报错内容作为验收证据。
3. 针对运行时失败执行自动修复 (最大重试: 2 次)。
4. 自动模式 -> 决策下一步动向。手动模式 -> 等待人工确认放行。

[质量底线]
- [必须] 所有验证决策只能基于真实的执行日志、断言结果等硬核证据。
- [禁止] 毫无凭证的口头推断式通过 (例如 AI 说出“理论上代码没问题”)。
- [必须] 测试用例必须独立放置在 `tests/` 目录下，并配置专属 `CMakeLists.txt`。
- [必须] 采用基于风险倾斜的覆盖率策略 (重点猛攻状态机流转与边缘边界值，而非追求流水账代码覆盖)。

[升级人工条件]
- 缺乏充足的执行日志或输出证据来支撑最终验收结论。
- 运行时报错在经过 2 次自动修复重试后仍未解决。
- 宣告整个系统任务的最终完成 (此环节绝不允许自动跳过，必须人工签字放行)。
